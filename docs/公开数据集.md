# 公开数据集

本项目不把大规模数据原文直接放进仓库，而是提供导入和转换脚本。这样做有三个原因：数据体积大、许可证可能变化、面试时更容易讲清楚“数据来自哪里、如何清洗、如何复现”。

## 推荐数据源

| 数据源 | 规模与特点 | 用法 |
| --- | --- | --- |
| JDDC | 论文介绍为超过 100 万段多轮中文电商客服对话、约 2000 万 utterances | 适合构建多轮客服 SFT 数据 |
| JDDC 2.1 | GitHub/论文介绍为约 24.6 万会话、300 万 utterances、50.7 万图片，并包含商品知识库和图片类别标注 | 文本部分可用于客服回复训练，多模态部分后续可扩展 |
| EcomGPT / EcomInstruct | AAAI 论文介绍 EcomInstruct 约 250 万条电商指令数据；ModelScope 页面介绍包含 122 个训练任务/数据集 | 适合补充电商指令任务和商品理解任务 |
| Hugging Face 电商客服/QA 数据 | 数据格式差异较大，需要字段映射 | 适合快速 smoke test 和小规模实验 |
| 自构 SOP 样本 | 覆盖物流、退货、发票、投诉、安全拒答等业务规则 | 适合增强规则遵循和安全边界 |

## 导入方式

Hugging Face 数据集：

```powershell
uv run python -m ecomcarelm import-hf --dataset DATASET_NAME --split train --question-field question --answer-field answer --output data/processed/public_sft.jsonl --limit 10000
```

本地 JSONL 数据：

```powershell
uv run python -m ecomcarelm build-sft --input data/raw/your_dataset.jsonl --output data/processed/sft_train.jsonl
```

## 面试里怎么讲

可以这样说：

> 我没有把大规模公开数据直接提交到仓库，而是实现了数据导入和字段映射层。这样一方面尊重数据集许可证，另一方面也能复现完整流程。训练数据主要来自 JDDC/JDDC 2.1 这类真实中文电商客服对话，再用 EcomInstruct 或电商 QA 数据补充商品理解、评论理解和指令跟随能力，最后用自构 SOP 补齐退货、物流、投诉和安全拒答等规则边界。
