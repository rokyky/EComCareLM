## 一、项目总览

### 1. EComCareLM 一句话怎么介绍？

**🎤 面试回答**

一句话说：EComCareLM 是一个中文电商客服垂域模型对齐项目，核心是把数据构建、模型训练、偏好对齐和评测这四个环节串成一个完整闭环，并提出了一种叫 PGDM 的方法来提升 DPO 数据质量。

展开来讲，它不是一个直接上线的客服机器人，而是一条对齐管线。具体做了四件事：第一，数据层——把 JDDC、EcomInstruct 等公开数据集做字段映射、PII 脱敏、近重复去重，统一成内部 schema。第二，训练层——基于 Transformers 和 TRL 框架，支持 LoRA/QLoRA 做 SFT 微调，让模型先学会客服话术。第三，对齐层——用 DPO 做偏好对齐，但这里有一个关键设计：rejected 不是用模板负样本，而是优先用模型自己在评测集上的真实错误回答。更进一步，我们通过 PGDM 方法，把规则评测和 LLM Judge 的维度级分歧作为硬负样本挖掘信号，按策略严重度×分歧强度×标注置信度加权采样来构造 DPO 数据。第四，评测层——规则评测做快速回归，LLM Judge 做多维质检，分歧挖掘定位不一致样本，HRPVR 作为高风险场景的主指标。这四层形成迭代闭环。

**🎯 面试官视角**

**考察点：** 面试官想确认你对项目定位是否清晰——不是”调包跑 demo”，而是有设计取舍的系统工程。他要听你说得出”为什么这么拆”、”难点在哪里”。

**追问 1：你说 PGDM 是核心方法，它和传统 DPO 有什么区别？**
答：传统 DPO 的 rejected 要么是模板负样本，要么是模型在某个评测集上的错误回答。PGDM 更进一步——它不是把所有错误回答都当成 rejected，而是优先选那些”规则评测觉得不行但 LLM Judge 觉得行”或者反过来的样本，因为这些分歧样本最可能是 hard negatives。再加上 policy severity 加权，让高风险策略（退款、隐私）的错误回答被更多采样。

**追问 2：那你们怎么验证 PGDM 确实比普通 DPO 好？**
答：我们设计了 6 组消融实验：从 Base → SFT → Template-DPO → Badcase-DPO → PGDM-DPO（去掉某部分）→ PGDM-DPO（完整），用 HRPVR 做主指标对比。0.5B 上跑趋势，7B 上出正式结论。

**常见误区：** 一上来就说”我做了个客服大模型”——面试官追问上线指标你就被动了。正确的定位是”垂域对齐的方法验证平台”，不是产品。另一误区是把 PPO 包装成已完成工作，但项目里 PPO 只是路线检查器，不是可运行的训练入口。

**🔗 项目结合**

**核心入口：** `ecomcarelm/cli.py` 定义了 13 个子命令，覆盖数据导入→清洗→构建→训练→评测全链路。

**四层代码对应：**
- 数据层：`ecomcarelm/datasets.py` + `builders.py` + `cleaning.py`
- 训练层：`train/train_sft.py` + `train/train_dpo.py` + `train/train_grpo.py`
- 对齐层：`train/train_dpo.py`（DPO）+ `train/train_grpo.py`（GRPO）+ `ecomcarelm/builders.py`（PGDM 数据构建）
- 评测层：`ecomcarelm/evaluation.py` + `judge.py` + `disagreement.py` + `policy.py`

**数据流：** `import-hf` 导入 → `clean` 脱敏去重 → `build-sft` 构建 SFT → `train_sft.py` 训练 → `demo`/模型预测 → `eval` 规则评测 + `judge` LLM Judge → `label-policy` 策略标注 → `mine-disagreements` 分歧挖掘 → `build-pgdm-dpo` 构造 DPO 对 → `train_dpo.py` 训练 → `hrpvr` 出指标。

### 2. 为什么选电商客服？

**🎤 面试回答**

一句话说：选电商客服是因为它有明确的业务规则和 SOP，适合把业务质量转成可计算、可验证的训练和评测信号。

展开来说有三个层面的考虑。第一，可评测性——退货流程、物流追踪、发票开具、投诉处理、安全拒答，每个子场景都有明确的对错标准。比如”退货退款必须在订单完成前申请”，这是一个可编程的规则，能写成 must_include 或 must_not_include 的检查项，放进规则评测和 GRPO reward 里。相比泛闲聊或开放域问答，电商客服的”对错”是可判定的，模型改进是可量化的。第二，对齐技术的天然试验场——客服需要同时满足多条约束：要准确、要礼貌、不能过度承诺、不能泄露隐私。这恰恰是对齐技术要解决的多目标优化问题。单一指标的 SFT 搞不定，所以需要 DPO 做偏好对齐、GRPO 做规则约束、LLM Judge 做多维质检。第三，业务价值清晰——客服是电商平台的核心触点，回答质量直接影响用户体验和合规风险，这个方向本身有真实的工程需求。所以选这个场景不是因为”简单”，恰恰是因为它够复杂，适合验证对齐方法。

**🎯 面试官视角**

**考察点：** 他想知道你是随手选了个场景，还是有意识地做了场景分析。说”电商客服数据好找”是不够的，要说”这个场景的约束结构适合做对齐技术验证”。

**追问 1：为什么不做金融客服或医疗客服？**
答：金融和医疗同样有强规则，但数据获取门槛更高，合规风险更大。电商客服有 JDDC、EcomInstruct 等公开数据集，适合在资源受限时先验证方法。如果资源充足，金融客服同样是很好的对齐场景，甚至可以复用 EComCareLM 的框架，只要换 policy taxonomy 和评测规则就行。

**追问 2：你觉得电商客服最难的对齐问题是什么？**
答：边界模糊的过度承诺。比如用户问”我一定能拿到赔偿吗”，规则说要”审核后处理”，但用户情绪激动时模型容易被带偏，给出肯定答复。这种场景下 rule-based reward 很难精确捕捉，需要 LLM Judge 和人工抽检兜底。

**常见误区：** 有人说选电商客服是因为”简单”——恰恰相反，电商客服的规则数量多、冲突多、变化快，比单一领域的客服更难做好。另外不要把这个项目说成”可以做生产级客服”，要强调它是”对齐技术的方法验证平台”。

**🔗 项目结合**

**场景拆分：** `ecomcarelm/builders.py` 中 `build_sft_records` 通过 `_compose_input` 接收 question/context/policy 字段，场景分类在数据导入时就打好了（`scenario` 字段）。

**规则定义：** `train/rewards.py` 中的 `客服规则奖励` 函数用 礼貌词、风险表达、must_include/must_not_include 实现多维打分，每个场景的规则集在 `must_include` 和 `must_not_include` 里显式定义。

**评测维度：** `ecomcarelm/evaluation.py` 的 `score_case` 函数根据 scenario、must_include、must_not_include、policy 等字段，输出 7 个维度的分数。

### 3. 为什么不是 RAG 方案？

**🎤 面试回答**

一句话说：RAG 解决的是”怎么查到最新知识”，而我们解决的是”怎么稳定、合规地说出来”，这是两个正交的问题，不能互相替代。

展开来讲，做一个客服机器人，如果只上 RAG，会碰到三类 RAG 解决不了的问题。第一，语言质量和话术——RAG 能把规则文档检索出来拼进 prompt，但不能保证模型用礼貌、完整、符合品牌调性的语言表达。模型可能语气生硬、遗漏关键安抚语——这些不是”有没有知识”的问题，是”会不会说话”的问题，需要 SFT 来学。第二，规则冲突和偏好——同一个问题可能匹配多条规则，比如用户投诉，模型既要安抚情绪，又要引导正式流程，还不能承诺处理结果。模型需要在多条约束之间做权衡，这不是检索能解决的，需要 DPO 或 GRPO 来做偏好对齐。第三，安全边界——RAG 给了模型更多上下文，但模型仍然可能忽略安全指令，自行编造赔偿方案。这类 badcase 需要通过训练让模型学会”哪些话绝对不能说”。所以我们的做法是：模型的对齐能力通过训练固化，动态规则通过 RAG 或规则引擎在推理时注入，两者配合而不是二选一。

**🎯 面试官视角**

**考察点：** 面试官想确认你不是”只会微调”或”只会 RAG”，而是理解两者的优缺点和适用边界。

**追问 1：如果规则每天变，你怎么办？**
答：稳定的话术和安全边界用微调固化；每天变的具体规则（如促销政策、运费标准）走 RAG 或规则库注入 prompt。这也是训练脚本里把 policy 作为独立输入字段的原因——它预留了外部知识注入的接口。

**追问 2：RAG 加微调一起上，效果一定是叠加的吗？**
答：不一定。如果微调数据里包含了过时的规则，RAG 给了最新规则，模型会产生内部冲突。需要设计实验验证 RAG 上下文的优先级是否覆盖了微调记忆，这也是评测里要单独测的维度。

**常见误区：** 很多人说”RAG 成本低所以更好”——实际上 RAG 的检索质量、上下文窗口管理、延迟优化都是工程挑战。微调和 RAG 不是替代关系，是不同层面的优化目标。

**🔗 项目结合**

**看代码：** `ecomcarelm/builders.py` 中的 `_compose_input` 函数为每条样本预留了 policy 字段位置——这就是未来接 RAG 的接口。当前 policy 是写死在数据里的，上线场景下 policy 可以从规则库实时检索注入。

**与 bitalab 的配合：** bitalab 的 MCP 工具编排方案适合做动态规则查询的工具调用，EComCareLM 训练出的模型作为 Agent 的底层 LLM，两者结合就是”领域特化 Agent”。

### 4. 和 bitalab 有什么区别？

**🎤 面试回答**

一句话说：bitalab 是 Agent 平台工程，解决怎么编排工具、管理对话、调度资源；EComCareLM 是模型能力工程，解决怎么让模型在垂域里更准确、更合规、更安全。

展开来讲，两个项目覆盖了面试中最常考察的两个维度。bitalab 那一端，涉及 MCP 协议、工具注册与发现、Multi-Agent 编排、Memory 管理、Runner 隔离、Quick Session 免登录设计、GPU 资源调度——这些都是系统架构层面的能力，证明我能搭一个可用的 Agent 平台。EComCareLM 这一端，涉及数据构建、洗数据、LoRA/QLoRA 训练、DPO 偏好对齐、GRPO 规则强化、规则评测和 LLM Judge——这些是模型算法层面的能力，证明我能做垂域模型的对齐和评测。两个项目加起来，能覆盖 Agent 开发岗里”会搭系统”和”懂模型”的两端。面试时我一般这样定位：如果有人问系统架构，我讲 bitalab 的设计取舍；如果有人问模型训练和对齐，我讲 EComCareLM 的数据闭环和训练管线。

**🎯 面试官视角**

**考察点：** 面试官想确认你不是只干过一件事，而是有广度。他也会观察你能不能把两个项目说清楚而不是混在一起。

**追问 1：如果两个项目合在一起会是什么？**
答：bitalab 提供 Agent 框架，EComCareLM 提供垂域模型能力，合在一起就是一个”电商客服特化 Agent”。用户在 bitalab 平台上创建会话，背后的客服模型跑的是 EComCareLM 微调过的版本，遇到需要查规则的场景再用 MCP tool 去检索。这是两个项目的自然延伸。

**追问 2：你觉得哪个项目更难？**
答：难度维度不同。bitalab 的难点在系统设计和工程落地——会话隔离、工具安全、资源调度，出 bug 就是线上事故。EComCareLM 的难点在实验设计和效果验证——数据怎么构造、reward 怎么设计、评测怎么可信，搞错了方向训练跑再多也没用。

**常见误区：** 千万不要把两个项目混在一起讲，面试官会认为你对两个的理解都不深。面试官问到哪个就展开哪个，只在被问到”有没有做过相关的东西”时再提另一个。

**🔗 项目结合**

**各自对应：**
- bitalab：MCP 工具注册、Quick Session 设计、Runner 生命周期管理、Memory 长短记忆
- EComCareLM：`datasets.py` → `train_sft.py` → `train_dpo.py` → `evaluation.py` 的训练评测管线

**面试分层准备：** Agent 开发岗面试，先讲 bitalab 再补充 EComCareLM 做模型背景；算法岗面试，先讲 EComCareLM 再提 bitalab 做工程能力背书。

### 5. 被问”是不是只是调包”怎么回应？

**🎤 面试回答**

一句话说：框架用成熟实现是合理的，项目的核心价值不在于写了多少框架代码，而在于用这些工具把垂域对齐的完整闭环搭建起来，并在每个环节做了有数据支撑的设计决策。

展开来讲，我不回避用 Transformers 和 TRL——它们是 PyTorch 生态的事实标准，不需要也不应该重复造轮子。但”调包”和”搭系统”之间有一个本质区别：调包是调用一个函数就结束了，搭系统是要把数据构建、训练、评测、迭代四个环节串起来，每个环节都有设计和取舍。具体来说：数据层面，多个公开数据集的字段格式完全不同，需要设计统一的 schema 映射和字段适配器，训练框架不替你处理这个问题。训练层面，LoRA target_modules 选哪些层、DPO 的 beta 怎么调、GRPO 的 reward 怎么组合——这些不是调一个 trainer.fit() 能决定的，需要理解模型结构和训练原理。评测层面，规则评测的 must_include/must_not_include 怎么写、LLM Judge 的 rubric 怎么设计才能避免 judge 本身的偏差——这些都需要自己设计和迭代。更进一步，我们提出了 PGDM 方法，把规则-Judge 分歧转化为 DPO 训练信号——这已经超出了”调包”的范畴，是方法论层面的创新。

**🎯 面试官视角**

**考察点：** 这是一个压力测试，不是真的质疑你的技术能力。面试官想看你是否有工程自信——碰到质疑是慌还是能冷静拆解。

**追问 1：那你有没有给 Transformers 或 TRL 提过 PR？**
答：如果提过就说提过，没有就说：当前重点是验证垂域对齐的方法论，项目稳定后会反向贡献社区。诚实但展示方向感。

**追问 2：不用 TRL，你自己实现 DPO loss 要多久？**
答：DPO loss 的核心就几行——算 policy 和 reference 的 logprob，做差，过 sigmoid。自己实现不难，但要在分布式训练、mixed precision、gradient checkpointing 里稳定跑，需要做的工程工作不止 loss 本身。TRL 的价值不在 loss 实现，而在训练管线的稳定性。

**常见误区：** 最差的回应是说”我没调包，全都自己写了”——面试官一听就知道不可能。反过来直接认怂”对，就是调包”也不行。正确的策略是承认工具、强调设计，把话题引导到设计决策上。

**🔗 项目结合**

**看代码说服自己：** 打开 `train/train_sft.py`，看 TrainingArguments 的配置项——每一个参数背后都涉及显存、速度、效果的权衡。面试官要听的不是你设了哪些参数，而是你为什么这么设。

### 6. 项目当前最大不足是什么？

**🎤 面试回答**

一句话说：最大不足是还需要在真实规模的数据上跑完完整的对比实验，拿到 Base/SFT/DPO/GRPO 各阶段的可量化指标，特别是 PGDM-DPO vs 其他 DPO 变体的 HRPVR 对比。

展开来讲，当前项目有完整的数据管线、训练入口和评测闭环，链路上是通的。但样例数据只是 smoke test 规模——训练集 3000 条，评测集 500 条——跑出来的指标只能说明管线能跑，不能说明模型效果好。一个完整的实验应该包括：1-3 万条 SFT 数据做基础微调，1-3 千条人工校验过的 DPO 偏好数据做偏好对齐，500-1000 条覆盖多场景、多难度的评测集，然后用同一组解码参数在 Base、SFT、Template-DPO、Badcase-DPO、PGDM-DPO、PGDM-DPO（完整）六个阶段上跑对比。指标上要重点对比 HRPVR、规则遵循率和偏好胜率。这个不足我在 README 和项目介绍里都写明了，没有提前编造指标。

**🎯 面试官视角**

**考察点：** 面试官想看你对自己项目的认知是否客观。说”没有不足”显得不成熟，说”全都是不足”又显得没重点。关键是能指出一个具体的、有改进方向的不足。

**追问 1：补齐指标之后，最大的风险是什么？**
答：最大的风险是 PGDM 的增益没有预期的大，可能是因为 JDDC 数据是弱标注、没有 gold standard 的原因。这不是失败，正好说明需要更深入的 badcase 分析和更多的数据工程。

**追问 2：如果现在面试，你给这个项目打几分？**
答：作为方法论验证和代码实现，80 分——管线完整、设计有取舍、代码可复现。作为有说服力的简历项目，还差真实指标，60 分。

**常见误区：** 一种是把项目吹太大，说”已经训练完达到 SOTA”——面试官追问细节就穿帮。另一种是把项目说得一文不值——面试官会觉得你没有工程判断力。正确的做法是诚实但正面：指出当前阶段、明确下一步、说明你知道怎么做好。

**🔗 项目结合**

**具体哪里不足：** `train/train_sft.py` 中 `--max-steps` 参数默认是 -1（按 epoch 跑），但当前 `--num-train-epochs` 默认是 1.0。正式实验需要改成 `--num-train-epochs 3` 并配合完整数据集。`data/samples/` 下的 eval_set_v1.jsonl 只有 10 条样本，正式实验需要几百到一千条。

### 7. 项目的核心贡献是工程还是方法？

**🎤 面试回答**

一句话说：两者都有——工程上搭建了完整可复现的对齐管线，方法上提出了 PGDM（Policy-Grounded Disagreement Mining）来改进 DPO 数据质量。

### 8. PGDM 是什么意思？一句话说清

**🎤 面试回答**

一句话说：PGDM 的全称是 Policy-Grounded Disagreement Mining——基于策略归因的评测分歧挖掘，核心是把规则评测和 LLM Judge 的维度级分歧转化为 DPO 训练信号。

展开来讲，这里有三个关键词。第一，”Policy-Grounded”——分歧不是泛泛而谈的”这个回答不好”，而是归因到具体的客服策略：退款策略违规、隐私泄露风险、投诉升级失败等。我们定义了 6 类策略，每类有 severity 和 violation_hints。第二，”Disagreement”——分歧指规则评测和 LLM Judge 在 7 个维度上的评分差异。有的样本规则评测打了满分但 LLM Judge 发现了问题，这就是 disagreement。我们的核心洞察是：分歧样本比一致样本更有训练价值，因为它暴露了单一评测视角无法捕捉的问题。第三，”Mining”——从所有评测样本中自动挖掘分歧样本，按 severity × disagreement_strength × confidence 加权排序，用 per-policy quota cap 控制采样分布，最终构造 DPO 偏好对。简单说，PGDM 就是让两个评测器互相”吵架”，从吵架最凶的样本里学习。

**🎯 面试官视角**

**考察点：** 面试官想确认你能否在两三句话内把一个新方法的核心逻辑讲清楚。这是”电梯演讲”能力。

**追问 1：这个方法和 active learning 有什么关系？**
答：思路有相似之处——都想找到最有信息量的样本。但 active learning 通常基于模型不确定性采样，PGDM 基于两个评测系统的分歧采样，而且分歧归因到具体策略维度，可解释性更强。

**追问 2：为什么叫”Policy-Grounded”而不是”Disagreement-Grounded”？**
答：因为分歧本身不够——同一个分歧出现在退款场景和礼貌场景，训练优先级完全不同。Policy grounding 让我们能按业务风险加权，这是区分 PGDM 和普通分歧采样的关键。

**常见误区：** 把 PGDM 简单理解为”两个评测器取不同结果的就留下来”——没有理解策略归因和 weighted sampling 才是核心。另一个误区是觉得 PGDM 只是 DPO 数据构建技巧——它实际上是一个完整的 badcase 发现→归因→加权→训练迭代方法论。

2. `ecomcarelm/evaluation.py` → 规则评测打分
3. `ecomcarelm/judge.py` → LLM Judge 打分
4. `ecomcarelm/policy_labeler.py` → 违规标注
5. `ecomcarelm/disagreement.py` → 维度级分歧挖掘
6. `ecomcarelm/builders.py` → `build_pgdm_dpo_records` → 加权采样构造 DPO 对
7. `ecomcarelm/policy.py` → `high_risk_policy_violation_rate` → HRPVR 指标验证

## 二、PGDM 核心方法

### 11. PGDM 的全称和核心思路是什么？

**面试回答**

一句话说：PGDM 的全称是 Policy-Grounded Disagreement Mining，核心思路是把规则评测和 LLM Judge 对同一个回答的评分分歧，转化为 DPO 训练信号，并按策略重要性加权采样。

展开来讲，PGDM 的核心洞察是：规则评测和 LLM Judge 对同一个客服回答往往有不一致的评分——规则评测说"这个回答合规"，但 LLM Judge 说"有问题"，或者反过来。这些不一致的样本恰恰是最有价值的训练数据，因为它们暴露了单一评测视角无法捕捉的边缘情况。PGDM 的做法分三步：第一，把规则评测和 LLM Judge 的输出在 7 个维度上做归一化对齐，逐维度计算分歧 delta；第二，把每个分歧归因到具体的客服策略（退款、隐私、投诉等），得到 severity；第三，按 severity × disagreement_strength × label_confidence 加权排序，用 per-policy quota cap 控制采样分布，构造 DPO 偏好对。

**面试官视角**

**考察点：** 面试官想确认你是否理解 PGDM 不是"随便找两个评测器比一比"，而是有设计意图的——为什么 disagreement 是好信号？为什么需要 policy grounding？

**追问 1：分歧挖掘和普通的 hard negative mining 有什么区别？**
答：hard negative mining 通常基于模型置信度或 loss，PGDM 基于两个评测系统的分歧，而且分歧被归因到具体策略维度，可解释性更强。

**追问 2：如果两个评测器都错了怎么办？**
答：PGDM 不假设评测器绝对正确。但规则评测是确定性的（关键词匹配），LLM Judge 有业务知识。当两者一致时可能正确，当两者分歧时至少有一方捕捉到了问题，值得模型重点学习。

**常见误区：** 把 PGDM 理解为"两个评测器结果不同就当成 DPO 数据"——忽略了 policy grounding 和 weighted sampling 才是让 PGDM 有效的原因。

**项目结合**

**核心代码流：** evaluation.py（规则评测）→ judge.py（LLM Judge）→ policy_labeler.py（策略归因）→ disagreement.py（分歧挖掘）→ builders.py 的 build_pgdm_dpo_records（加权采样）

**设计文档：** 看 openspec/changes/add-pgdm-dpo/design.md，有完整的数据流图和 weight 计算公式。


### 12. 为什么规则评测和 LLM Judge 的分歧是有价值的训练信号？

**面试回答**

一句话说：因为分歧暴露了单一评测视角的盲区，这些样本恰恰是模型最需要学习的"模糊边界案例"。

展开来讲，规则评测基于关键词匹配，优点是确定性强、可复现、零成本；缺点是它不懂语义——比如"建议不要私下转账"包含了"私下转账"这个关键词，但实际是安全提醒。LLM Judge 懂语义，但它有模型偏好、prompt 敏感、位置偏差等问题。当一个样本让这两个评测器得出不同结论时，说明这个样本至少处于一个或两个评测器的边界盲区里——这正是高质量的训练信号。具体到客服场景，最常见的分歧类型是"rule_pass_judge_fail"——规则评测说回答安全，LLM Judge 发现过度承诺。这种分歧暴露了模型"看起来合规但实际有问题"的 badcase。

**面试官视角**

**考察点：** 面试官想确认你理解为什么 disagreement 比 agreement 更有信息量——这是 PGDM 的理论基础。

**追问 1：那 agreement 样本是不是就没用了？**
答：不是没用，但信息量更小。两个评测器一致认为好或坏，说明模型在这个样本上已经稳定了。分歧样本才是模型尚未稳健的区域。

**追问 2：分歧的 direction 有区别吗？**
答：有。rule_pass_judge_fail 更危险——规则评测放行但 LLM Judge 发现问题，说明模型有"隐蔽违规"的问题，客服场景里最应该优先修复。

**常见误区：** 认为 agreement 就是"真理"，disagreement 是"噪声"——两个评测器都不完美时，agreement 可能只是共有盲区。

**项目结合**

**代码：** disagreement.py 的 _direction 函数定义了 4 种 direction。build_pgdm_dpo_records 的 metadata 记录 aggregate_disagreement。


### 13. 维度级分歧挖掘怎么对齐规则和 Judge 维度？

**面试回答**

一句话说：把规则评测和 LLM Judge 的输出分别归一化到 7 个共享维度上，逐维度计算分数差，超过阈值的记为维度级分歧。

展开来讲，规则评测输出 7 个维度（answer_accuracy、policy_compliance、completeness、politeness、safety、forbidden_content、off_topic），LLM Judge 输出 6 个评分项（accuracy、completeness、politeness、safety、hallucination、off_topic）。它们不是一一对应的——规则评测的 accuracy 是综合分数，LLM Judge 的 accuracy 映射到 policy_compliance；Judge 的 hallucination 在规则评测里没有对应；规则评测的 forbidden_content 在 Judge 里也没有。对齐规则是：只在双方都拥有的维度上做比较，单方独有的静默跳过。对齐后对每个共享维度算 delta = |rule_score - judge_score|，如果 delta >= threshold（默认 0.35），记录一条维度级分歧。同时有 aggregate 级别：7 个维度的平均分是否超过 pass_threshold（默认 0.75）。

**面试官视角**

**考察点：** 面试官想知道两个评测器格式完全不同，你怎么对齐——这是 PGDM 最工程细节的部分。

**追问 1：0.35 阈值怎么定的？**
答：经验值。太小引入噪声分歧，太大漏掉边缘案例。正式实验时应跑阈值扫描。

**追问 2：hallucination 和 forbidden_content 为什么不对齐？**
答：它们是不同概念——hallucination 是编造规则/事实，forbidden_content 是包含违禁词。强行对齐会引入噪声。AI_REVIEW_GUIDE 里记录了这是 design decision。

**常见误区：** 以为两个评测器的维度可以直接比较——格式完全不同，需要先 normalize 再 map。

**项目结合**

**核心函数：** disagreement.py 的 normalize_rule_dimensions 和 judge.py 的 normalize_judge_dimensions。judge.py 中 accuracy 映射到 policy_compliance。


### 14. 分歧挖掘输出 aggregate 和 dimension 两级，各什么含义？

**面试回答**

一句话说：aggregate 是整体级别的分歧判断（both_pass/both_fail/rule_pass_judge_fail/rule_fail_judge_pass），dimension 是每个维度上的详细分歧记录。

展开来讲，aggregate_disagreement 计算 7 个维度的平均分，根据是否超过 pass_threshold 分成四种情况。后两种是真正的分歧——rule_pass_judge_fail 和 rule_fail_judge_pass，PGDM 重点关注。dimension_disagreements 是每个维度的细粒度记录——维度名、双方分数、delta、direction。两级用途不同：aggregate 快速筛选分歧样本，dimension 用于精细加权——PGDM 的 disagreement_strength 从 dimension_disagreements 中取最大 delta 计算。

**面试官视角**

**考察点：** 确认你不只是调了接口，而是理解了两级设计的意图。

**追问 1：both_pass 和 both_fail 的样本怎么处理？**
答：both_pass 不会成为 DPO 候选，both_fail 也不是最高价值候选。Gold mine 是 one-pass-one-fail。

**追问 2：missing_judge 说明什么？**
答：没有对应的 Judge 结果。此时 disagreement_strength floor=1.0，保证不被完全丢弃。

**常见误区：** 只关注 aggregate 不看 dimension——两个评测器整体 pass/fail 相同，但可能某个维度剧烈分歧。

**项目结合：** disagreement.py 的 compare_dimensions。builders.py 的 _best_dimension 取最大 delta。


### 15. PGDM-DPO 的 sample_weight 为什么是 severity x disagreement_strength x confidence？

**面试回答**

一句话说：这个乘积式把"策略风险"、"分歧强度"和"标注可信度"三个正交信号组合成一个统一的采样权重。

展开来讲，severity 解决"不同策略违规代价不同"——退款违规 severity=1.5，隐私泄露 1.8，语气问题 0.8。disagreement_strength 解决"分歧大小不同"——delta=1.0 的分歧比 delta=0.3 的信息量更大。confidence 解决"标注来源可信度不同"——规则命中 confidence=1.0，LLM 高置信 0.8，unknown 0.2。三个相乘，三项都在高位的样本优先进入 DPO 训练集。

**面试官视角**

**考察点：** 确认 weight 不是拍脑袋定的，每个因子都有明确的业务和技术含义。

**追问 1：乘法会不会让某个因子主导？**
答：会，这是有意为之。Per-policy quota cap 防止某类策略垄断。

**追问 2：为什么不用加法？**
答：加法让各因子可以互相补偿——低 severity 的样本用高 disagreement 也能到高 weight。乘法更严格。

**项目结合：** builders.py 的 build_pgdm_dpo_records 第 181 行。policy.py 的 POLICY_DEFINITIONS 定义 severity。


### 16. policy_cap 是什么？为什么需要 per-policy quota cap？

**面试回答**

一句话说：policy_cap 是每类策略在 PGDM-DPO 中能占的最大比例，防止高风险策略垄断 DPO 训练集。

展开来讲，refund_policy severity=1.5，privacy_policy=1.8，tone_policy=0.8。不加限制时高风险样本占大部分，模型只在退款和隐私场景优化，忽视其他场景。policy_cap 解决这个问题——target=1000，cap=0.5，每类最多 500 条。

**面试官视角**

**考察点：** 确认你注意到 weighted sampling 可能导致数据倾斜。

**追问：policy_cap 设多少？**
答：默认 0.5。6 类策略时 0.5 宽松，3 类时可设 0.7。

**常见误区：** 以为 cap 是硬限制——实际上是上限，不会强制填满。

**项目结合：** builders.py 的 build_pgdm_dpo_records 第 211-225 行。CLI 参数 --policy-cap。


### 17. HRPVR 是什么？为什么选它做主指标？

**面试回答**

一句话说：HRPVR 是高风险策略违规率，只统计退款、隐私、投诉升级三类高风险策略上模型回答的违规比例。

展开来讲，统计所有归因为 refund_policy、privacy_policy、escalation_policy 的样本，检查回答是否违反对应策略——policy_compliance 低于阈值、safety 低于阈值，或 badcase_type 属于 over_promise_or_unsafe、policy_conflict、forbidden_content。选它做主指标的两个原因：第一，聚焦高风险场景——退款、隐私、投诉最容易出问题；第二，比整体指标更敏感——整体指标把高低风险混在一起，高风险改进容易被稀释。

**面试官视角**

**考察点：** 确认主指标选择有道理。

**追问：HRPVR 谁算的？**
答：当前是规则评测算的，保证确定性和可复现性。分歧挖掘阶段才用 LLM Judge。

**常见误区：** 以为 HRPVR 越低越好——太低可能说明评测集太简单或模型过于保守。

**项目结合：** policy.py 的 high_risk_policy_violation_rate。HRPVR_RELEVANT_BADCASES 只包括三类。


### 18. HRPVR 只计哪三类 badcase？为什么排除 incomplete 和 impolite？

**面试回答**

一句话说：只计 over_promise_or_unsafe、policy_conflict、forbidden_content 三类，因为它们直接对应高风险策略违规；incomplete 和 impolite 虽有问题但不属于核心违规。

展开来讲，over_promise_or_unsafe 涉及赔偿承诺、违规协助等高危行为；policy_conflict 说明回答和平台规则矛盾；forbidden_content 是违禁表达。incomplete 不直接构成高风险违规——说了"可以申请"但没说"需上传照片"，不完整但不算违规。impolite 同理。排除是为了保持 HRPVR 的精度。

**面试官视角**

**考察点：** 确认你理解 HRPVR 故意只覆盖"红线问题"。

**追问：同时有 incomplete 和 over_promise 怎么算？**
答：classify_badcase 优先级——safety 高于 completeness，记为 over_promise_or_unsafe，计入 HRPVR。

**项目结合：** policy.py 的 HRPVR_RELEVANT_BADCASES。evaluation.py classify_badcase 的优先级。


### 19. 6 组消融实验分别对比什么？

**面试回答**

6 组：① Base（原始模型）→ ② SFT（标准客服 SFT）→ ③ Template-DPO（模板负样本）→ ④ Badcase-DPO（模型预测错误）→ ⑤ PGDM-DPO（去掉策略加权）→ ⑥ PGDM-DPO（完整）。预期 SFT 显著优于 Base，Template-DPO 略优于 SFT，Badcase-DPO 优于 Template-DPO，PGDM-DPO（完整）优于所有组。第⑤组验证策略加权是否必要。

**面试官视角**

**考察点：** 确认你理解每个实验对应一个技术假设。

**追问：没有 GRPO？**
答：GRPO 是另一条路线和 DPO 正交。当前只对比 DPO 的 rejected 构造策略。

**项目结合：** README.md 的表格。dpo_template / dpo_badcase / dpo_pgdm 三个数据文件。


### 20. PGDM-DPO 和 Template-DPO 的 rejected 本质区别？

**面试回答**

Template-DPO 的 rejected 是写死的模板字符串（五类模板），PGDM-DPO 的 rejected 是模型自己在评测集上的真实错误回答，经分歧筛选和策略加权。模板和模型实际错误脱节，PGDM 直接针对模型当前短板。

**面试官视角**

**追问：Template-DPO 一点用没有？**
答：冷启动时可用，模型有一定能力后模板梯度消失。

**项目结合：** builders.py 的 NEGATIVE_TEMPLATES。PGDM 的 rejected 来自 predictions。


### 21. PGDM-DPO 和 Badcase-DPO 的 rejected 本质区别？

**面试回答**

Badcase-DPO 用所有模型错误回答做 rejected，PGDM-DPO 只选用分歧样本加权。Badcase-DPO 没筛选，可能包括 harmless 措辞差异。PGDM 选"最有争议的错误"优先学。

**面试官视角**

**追问：Badcase-DPO 引入噪声？**
答：会——harmless 措辞差异不应作为 rejected。

**项目结合：** datasets.py 的 build_dpo_from_predictions。builders.py 的 build_pgdm_dpo_records 需要额外输入 policy_labels 和 disagreements。


### 22. PGDM 为什么需要 policy labeler？

**面试回答**

为每个分歧样本归因到具体客服策略，sample_weight 中 severity 的来源。Rule labeler 通过关键词匹配，precision 高、recall 低；LLM labeler recall 高但有成本。当前 rule labeler 作为 MVP，接口已兼容 LLM labeler。

**面试官视角**

**追问：recall 低会漏掉吗？**
答：会，但分歧挖掘和规则评测可以补偿。

**项目结合：** policy_labeler.py。policy.py 的 POLICY_DEFINITIONS。


### 23. disagreement_strength 的 floor 为什么是 1.0？

**面试回答**

Judge 结果缺失时 strength 设为 1.0，保证样本不被零权重排除也不被过度加权。missing_judge 不等于 no_disagreement。

**项目结合：** builders.py 的 _disagreement_strength 函数。


### 24. 如果规则评测和 LLM Judge 完全一致，PGDM 还有用吗？

**面试回答**

分歧挖掘层无信息增益，但策略加权层仍有效——退化为带加权的 Badcase-DPO。实际中几乎不可能，规则评测（关键词）和 LLM Judge（语义）有本质差异。

**面试官视角**

**追问：怎么验证分歧信号不是噪声？**
答：消融实验⑤组（只用分歧）vs ④组（Badcase-DPO）。

## 三、数据构建

### 25. 公开数据怎么接入？字段映射怎么设计的？

**面试回答**

一句话说：数据接入的核心不是下载，而是解决不同数据集的 schema 差异，统一成内部标准格式。

展开来讲，不同电商数据集结构差异很大。JDDC 是客服对话格式，EcomInstruct 是指令格式，其他数据集可能叫 question/answer 或 query/response。如果写死字段名，每接一个新数据集就要改代码。所以设计了字段映射机制：在导入时指定每个外部字段对应内部 schema 的哪个字段——question、answer、context、policy。经过映射后统一成 instruction/input/output 三元组。映射规则写在参数里而不是代码里，加新数据集只需改参数。

**面试官视角**

**考察点：** 确认你不是只会 load_dataset() 然后直接喂模型。

**追问：两个数据集字段名一样但含义不同？**
答：需要加 transform 预处理。当前方案允许在建数据时传入自定义 transform 函数。

**常见误区：** 说"用 Hugging Face datasets 的 map 统一处理"——但 map 是处理函数不是 schema 管理，不同数据集字段定义不同时 map 里全是 if-else。

**项目结合：** datasets.py 的 convert_records 做字段映射。builders.py 的 build_sft_records 消费映射后的统一格式。


### 26. JDDC 和 EcomInstruct 各适合什么场景？

**面试回答**

JDDC 解决"怎么在客服对话中得体回应"，EcomInstruct 解决"怎么理解电商特有的商品和交易语义"。JDDC 是客服真实对话，适合 SFT 和 DPO 做多轮售后训练。EcomInstruct 覆盖商品标题、属性提取、评论摘要等任务，适合补充电商语义理解。两类互补——JDDC 补对话能力，EcomInstruct 补商品理解能力。

**面试官视角**

**追问：JDDC 有什么已知问题？**
答：较老（2020），促销政策可能过时；客服回复不代表最优回答风格。做 DPO 的 chosen 时需规则评测筛选。

**项目结合：** builders.py 中 JDDC 和 EcomInstruct 走不同的构建分支。


### 27. JDDC 数据有哪些已知限制？

**面试回答**

JDDC 是真实客服对话，但没有人工精标的标准答案，只有客服原始回复作为"弱标注"参考。policy 字段为空——JDDC 没有规则标注。must_include 和 must_not_include 为空数组——_coverage(answer, []) 返回 1.0 满分。所以依赖 policy 的场景（如 policy_compliance）在 JDDC 上精度有限。HRPVR 等指标的绝对值仅供参考，应关注不同方法之间的相对趋势。

**项目结合：** README.md 的 JDDC 数据限制章节。eval_set_v1.jsonl 是自构数据，JDDC 导入的 eval 集缺乏 policy 和 must_include。


### 28. JDDC 多轮对话怎么拆成单轮 QA 对的？

**面试回答**

JDDC 原始格式是 TSV，每行一条消息含 waiter_send 字段（1=客服、0=用户）。脚本按 session_id 分组，遍历消息序列以最近的用户消息为 question、对应的首条客服回复为 answer 来构造单轮 QA 对。同时按关键词给每条 QA 打场景标签（return_refund、logistics 等），用于下游评测的分析。

**项目结合：** scripts/import_jddc_baseline.py。场景推断通过关键词匹配。


### 29. import_jddc_baseline.py 里的场景推断怎么做？

**面试回答**

基于关键词匹配——检测对话中出现"退货"、"退款"、"破损"归为 return_refund；"物流"、"快递"、"没到"归为 logistics；"投诉"、"举报"归为 complaint。匹配到多条时取优先级最高的。


### 30. PII 脱敏为什么重要？做了哪 6 类脱敏？

**面试回答**

客服数据天然包含隐私——手机号、订单号、快递单号、银行卡号、身份证号、地址都可能出现。模型如果记住这些内容，推理时可能泄露，违反《个人信息保护法》。当前做了 6 类正则脱敏：手机号、身份证号、订单号、快递单号、银行卡号、地址。用占位符替换（如 [PHONE]）。

**面试官视角**

**追问：脱敏后模型效果下降？**
答：可能。直接标识符必须脱敏，业务数字（价格、日期）需保留或格式化脱敏。

**项目结合：** cleaning.py 的 PLACEHOLDERS 列表。


### 31. 正则脱敏局限是什么？生产环境怎么办？

**面试回答**

正则只能覆盖固定模式——"137xxxx"这种部分脱敏写法正则不好处理。地址脱敏误伤率高，混合内容和方言表达难覆盖。生产环境应加 NER 模型做实体识别 + 人工抽检兜底。

**项目结合：** cleaning.py 当前用正则，注释标注了生产环境建议。


### 32. 去重阈值为什么用 0.92？

**面试回答**

0.92 是偏保守的阈值，在"保留差异"和"去除冗余"之间折中。太高（0.99）去不掉重复，太低（0.8）误删语义不同但措辞相近的样本。0.92 用 difflib 的 SequenceMatcher 做近重复检测。

**面试官视角**

**追问：数据量小还要去重？**
答：要，但应更保守——0.95 以上只去完全相同的。

**项目结合：** cleaning.py 的 clean_records 函数，dedup_threshold 是可配置参数。


### 33. 近重复去重和精确去重有什么区别？

**面试回答**

精确去重只去掉完全相同的两行，近重复检测用文本相似度找高度相似但不完全相同的样本。客服数据几乎没有两行完全相同的（用户 ID、订单号、日期不同），精确去重只能去文件级的重复。近重复去重才能找到语义重复但措辞不同的样本。

**项目结合：** cleaning.py 用 difflib.SequenceMatcher 而非 set 去重。


### 34. SFT 数据的 instruction/input/output 格式各是什么？

**面试回答**

instruction：系统提示词，如"你是中文电商平台客服..."
input：用户问题、上下文和平台规则拼接，包含 question / context / policy / product。
output：标准客服回答。

**项目结合：** builders.py 的 build_sft_records。train_sft.py 的 format_sft_text 进一步拼接成 chat template。


### 35. DPO 数据的 chosen/rejected 有几种构造方式？

**面试回答**

三种：Template-DPO——chosen 用 gold answer，rejected 用模板字符串。Badcase-DPO——chosen 用 gold answer，rejected 用模型自身在评测集上的错误预测。PGDM-DPO——chosen 用 gold answer，rejected 用经分歧筛选和策略加权的模型错误预测。

**项目结合：** build-dpo → builders.py build_dpo_records。build-dpo-from-predictions → datasets.py build_dpo_from_predictions。build-pgdm-dpo → builders.py build_pgdm_dpo_records。


### 36. build_dpo_from_predictions 怎么工作的？

**面试回答**

接收两个输入：eval_set（含 gold_answer）和 predictions（模型回答）。对每个 case，从 eval_set 取 gold_answer 做 chosen，从 predictions 取模型回答做 rejected。如果 chosen 和 rejected 文本相同则跳过。输出的 DPO pair 带 metadata 标注 source=model_prediction_badcase。

**项目结合：** datasets.py 的 build_dpo_from_predictions。


### 37. chosen 和 rejected 相同时为什么要跳过？

**面试回答**

DPO loss 要求 chosen 和 rejected 不同——如果两者相同，logprob 差为 0，loss 为 0，训练无效。这通常发生在模型已在该样本上回答正确时。

**项目结合：** 三个构建函数都有 "identical_pair" 跳过逻辑。


### 38. SFT 数据太模板化有什么风险？

**面试回答**

风险一：泛化能力差——措辞不同的同义问题时模型匹配不上模板。风险二：规则冲突时随机振荡——两个冲突模板在数据里都出现过。更隐蔽的是模板化让 eval 好看但上线崩。解决：难度分层、表达多样化、DPO/GRPO 纠偏。


### 39. train/dev/test 的切分比例和随机种子怎么设？

**面试回答**

默认 90%/5%/5%，seed=42。先 shuffle 再切分，确保三集不重叠。seed 固定保证幂等性——同一输入多次运行切分相同。

**项目结合：** datasets.py 的 split_records。CLI import-hf 的 --train-ratio/--dev-ratio/--seed。


### 40. JDDC 数据为什么没有 policy 和 must_include？对评测有什么影响？

**面试回答**

JDDC 是真实对话数据，不是人工标注数据。它没有规则标注（policy）和关键词标注（must_include）。所以评测时 policy_compliance 和 completeness 精度有限——coverage(answer, []) 永远 1.0 满分。所以 JDDC 评测只能看相对趋势，不能看指标绝对值。自构 SOP 数据才有人工标注的 policy 和 must_include。

**项目结合：** README.md 的 JDDC 限制说明。自构数据在 eval_set_v1.jsonl 里有 policy 和 must_include。

## 四、Policy Taxonomy & Labeling

### 41. Policy taxonomy 定义了哪 6 类策略？各自的 severity 是多少？

**面试回答**

6 类策略按 severity 从高到低：privacy_policy（1.8，高风险）、refund_policy（1.5，高风险）、escalation_policy（1.4，高风险）、evidence_policy（1.3）、logistics_policy（1.1）、tone_policy（0.8）。还有一个兜底的 unknown_policy（0.2）。

**面试官视角**

**考察点：** 确认 taxonomy 是按业务风险设计的，不是随便分的。

**追问：severity 怎么确定的？**
答：基于业务影响评估——隐私泄露的法律风险最高，退款违规次之，语气问题最轻。

**项目结合：** policy.py 的 POLICY_DEFINITIONS 字典。


### 42. 高风险策略是哪几类？为什么？

**面试回答**

refund_policy（退款违规）、privacy_policy（隐私泄露）、escalation_policy（投诉升级失败）。这三类直接影响平台合规风险和用户权益。退款违规涉及不当赔偿承诺；隐私泄露可能违法；投诉升级失败可能导致用户投诉渠道外溢。

**项目结合：** policy.py 的 HIGH_RISK_POLICY_IDS。


### 43. tone_policy 的 violation_hints 举例？为什么 severity 只有 0.8？

**面试回答**

violation_hints 包括"你自己"、"不关"、"没办法"、"别问"等。severity 0.8 说明语气问题不影响合规和安全，但影响用户体验——所以它不会进 HRPVR 统计，但会被规则评测的 politeness 维度捕捉。


### 44. policy_labeler 的 rule-based 标注流程是什么？

**面试回答**

对每个 case，先根据 scenario 推断对应 policy，然后把 question/context/policy/answer 拼接，按顺序检查：privacy → evidence → escalation → tone → refund。每组有专属 markers（如 OVER_PROMISE_MARKERS）。一旦命中就返回 policy_id、violation_type、confidence=1.0、evidence。都不命中则返回 unknown_policy/confidence=0.2。

**项目结合：** policy_labeler.py 的 label_policy_violations。


### 45. 过度承诺检查为什么在 unknown policy 时也要兜底？

**面试回答**

这是 BL-008 修复的核心。原 bug：scenario_policy==UNKNOWN 时，else 分支追加 (UNKNOWN, "over_promise", markers)，但循环第一行 if policy_id == UNKNOWN: continue 直接跳过了。修复后：用 refund_policy 兜底检查 over_promise，确保未知场景也被覆盖。

**项目结合：** policy_labeler.py 第 61-64 行的修复逻辑。BUG_LEDGER.md 的 BL-008。


### 46. policy label 的 confidence 怎么定的？unknown 为什么是 0.2？

**面试回答**

规则命中 1.0，LLM 高置信 0.8，LLM 低置信 0.5，unknown 0.2。unknown=0.2 表示"几乎没有归因信息"，在 PGDM weight 中是极弱信号，通常被 min_label_confidence（默认 0.5）过滤掉。


### 47. LLM-based policy labeler 比 rule-based 好在哪？

**面试回答**

LLM 可以理解语义上下文——比如"不要私下转账"会被 rule-based 误判为 escalation violation（因为含"私下转账"），但 LLM 能理解这是安全提醒。LLM 的 recall 更高但 precision 波动，有成本和延迟。


### 48. evidence 字段有什么用？

**面试回答**

记录触发违规判定的具体证据（如"['一定今天', '全额赔偿']"）。用于 debug——开发者可以看模型回答到底哪句话触发了违规。


### 49. infer_policy_from_scenario 怎么根据场景推断策略？

**面试回答**

遍历 POLICY_DEFINITIONS，检查每个 policy 的 scenario_hints 是否匹配场景名。例如 scenario="return_refund" 匹配 refund_policy 的 scenario_hints（'refund','return','after_sale'），返回"refund_policy"。无匹配返回 unknown。

**项目结合：** policy.py 的 infer_policy_from_scenario。


## 五、SFT 训练

### 50. SFT 解决什么问题？局限是什么？

**面试回答**

SFT 让模型学会任务格式和基础回答能力——客服应该怎么说话、怎么引用规则。局限：SFT 只能模仿答案，不能比较两个回答哪个更好，所以后面需要 DPO/GRPO 做偏好对齐和规则约束。

**项目结合：** train/train_sft.py。SFT 产出是 DPO 和 GRPO 的起点。


### 51. SFT 的 loss 怎么理解？

**面试回答**

Cross-entropy loss：对 answer 里每个 token，模型预测一个概率分布，loss = -log(p[target_token])，对所有 answer token 平均。目标就是让模型在"该说'您好'的地方说'您好'"。

**面试官视角**

**追问：为什么只在 answer 上算 loss？**
答：prompt 是输入条件，在 prompt 上算损失会让模型去"记住"用户问题。


### 52. SFT 为什么只对 assistant response 算 loss？prompt 部分怎么 mask？

**面试回答**

标准做法：prompt 部分的 labels 设为 -100，Transformers 在计算 loss 时会忽略 -100 的位置。实现上从后往前搜索 RESPONSE_MARKER（\n<|im_start|>assistant\n）的位置，marker 之前设 -100，之后保留原始 token ID。

**面试官视角**

**追问：为什么从后往前搜？**
答：防止 response 中恰好出现和 marker 相同的子串被误认。

**项目结合：** train_sft.py 的 tokenize 函数。BUG_LEDGER.md 的 BL-009。


### 53. LoRA 原理是什么？为什么不是全参微调？

**面试回答**

LoRA 冻结原始权重 W，在旁边加两个低秩矩阵 A（d×r）、B（r×d），用 BA 表示任务增量。训练时只更新 A 和 B。这样参数量、梯度和优化器状态都小很多。垂域适配不需要全参——LoRA 更快验证数据和对齐策略。全参微调成本高、容易破坏通用能力。

**项目结合：** train_sft.py 的 --use-lora 和 LoraConfig。


### 54. target_modules 为什么选 q/k/v/o 和 gate/up/down？

**面试回答**

q/k/v/o 决定模型注意力的范围和聚合方式——客服要读规则和上下文，适配这里影响模型的信息获取能力。gate/up/down 是 MLP 的主要表达通路——适配这里影响模型的回答表达能力。客服既要读懂规则也要会表达，所以两边都适配。


### 55. LoRA 的 rank/alpha/dropout 怎么选？

**面试回答**

rank 从 8/16 起步，控制 adapter 容量；alpha 通常和 rank 搭配（alpha=2×rank）；dropout 在数据少时防过拟合。默认 16/32/0.05。先小 rank 跑基线，再根据 badcase 决定是否加大。


### 56. QLoRA 和 LoRA 有什么区别？

**面试回答**

LoRA 省训练参数量，QLoRA 在此基础上进一步省底座权重的显存——把底座模型用 4bit NF4 量化加载，只训练 LoRA adapter。额外技术点：double quantization 量化量化参数本身，节省更多显存。

**项目结合：** train_sft.py 的 --load-in-4bit 和 BitsAndBytesConfig。


### 57. merge_and_unload() 为什么重要？

**面试回答**

LoRA 训练后 model 是 PeftModel，如果直接 save_pretrained 保存的是 adapter 权重而不是合并后的完整模型。DPO/GRPO 加载时基座权重缺失，训练失败。merge_and_unload() 把 adapter 合入基座权重，返回合并后的完整模型再保存。

**项目结合：** train_sft.py 第 179-181 行。BUG_LEDGER.md 的 BL-014。


### 58. gradient accumulation 解决什么？

**面试回答**

显存放不下大 batch 时，把一个大 batch 拆成多步小 batch，累积每个小 batch 的梯度，累积到指定步数后再更新一次参数。效果等价于大 batch，但显存占用只和单步小 batch 有关。


### 59. gradient checkpointing 原理是什么？

**面试回答**

训练时正向传播的中间激活占大量显存。Checkpointing 不保存部分中间激活，反向传播时重新计算。本质是计算换显存——省 50%+ 显存，增加约 20% 计算时间。长上下文训练时推荐打开。


### 60. mixed precision 有什么风险？bf16 vs fp16 怎么选？

**面试回答**

fp16 动态范围窄容易溢出。bf16 动态范围和 fp32 相同但精度低，训练更稳定。选型：支持 bf16 的 GPU（A100/4090 及以上）用 bf16，否则用 fp16。


### 61. 为什么训练前要做 smoke test？

**面试回答**

快速暴露字段缺失、tokenizer 错误、显存不足、依赖缺失问题。项目里 --max-steps 5 配合小样本验证链路是否通。避免跑了几小时发现数据格式有错。


## 六、DPO & 偏好对齐

### 62. DPO 原理直觉是什么？和 PPO 核心区别？

**面试回答**

DPO 直觉：不训练独立的 reward model，直接让 policy model 对 chosen 的 logprob 高于 rejected。PPO 需要训练 reward model、value model，在线采样更新。DPO 离线简单稳定，适合第一阶段对齐；PPO 在线但工程复杂。

**面试官视角**

**追问：DPO 没有 reward model 怎么做对齐？**
答：DPO 的 loss 隐含地在比较 policy 和 reference 对 chosen/rejected 的 logprob 差——差的增大表示模型偏离参考模型以更偏好 chosen。


### 63. DPO beta 是什么？beta 大小的影响？

**面试回答**

beta 控制偏好优化强度。beta 小≈优化激进，模型偏离参考模型远；beta 大≈更新保守，靠近参考模型。客服场景：beta 太小模型可能变啰嗦/过度拒答，太大效果不明显。从 0.1 起步调。

**项目结合：** train_dpo.py 的 --beta 参数。


### 64. 为什么 rejected 要来自真实 badcase？

**面试回答**

模板负样本和模型实际错误脱节。如果模型常犯"一定今天到账"的错误，rejected 就应该是这个错误。模板说"无法处理"而模型从不说这句话，DPO 的梯度方向就不对。

**项目结合：** datasets.py 的 build_dpo_from_predictions。


### 65. 模板负样本（Template-DPO）的根本问题是什么？

**面试回答**

模板语义固定，模型很容易学会区分模板和真实回答，从而不学习真正的偏好信息。更严重的是，模板可能和场景完全不相关（退款场景用 off_topic 模板）——训练信号是噪声。


### 66. DPO 训练后模型变啰嗦/变拒答怎么办？

**面试回答**

常见原因是 chosen 平均比 rejected 长，模型学到"长=好"。解决方案：控制偏好数据长度偏差、在评测中加入简洁性指标。

**项目结合：** 评测的 completeness 已覆盖，简洁性可额外加入。


### 67. PGDM-DPO 的 rejected 本质是什么？

**面试回答**

既不是模板也不是原始 badcase，而是经过分歧筛选和高风险加权的 badcase。核心：只选让规则评测和 LLM Judge 分歧最大的 badcase，再按策略 severity 加权。最终进入 DPO 训练集的是"最具信息量的错误"。


### 68. DPO 数据量太小怎么办？

**面试回答**

最少几百条有效 DPO 对就能看到信号。太少时优先确保数据质量而非数量——一条高质量分歧 badcase 胜过十条模板负样本。


### 69. DPO 失败模式有哪些？

**面试回答**

过拟合偏好（训练集指标好、通用能力下降）、过度拒答（模型什么都不回答）、模板化（所有输出风格趋同）。根本原因通常不是 DPO 算法本身，而是偏好数据质量差。


### 70. DPO 训练时 chosen 比 rejected 长很多有问题吗？

**面试回答**

有。DPO 会同时学到"长回答更好"的偏好。应使 chosen 和 rejected 长度大致匹配，或加长度正则。

**项目结合：** 评测包含 completeness 但不包含简洁性。正式实验时要关注。


## 七、GRPO & 规则奖励

### 71. GRPO 原理是什么？和 PPO 区别？

**面试回答**

GRPO 对同一 prompt 生成多条 completion，用组内相对 reward 优化——同组内得分高的回答被强化，低的被抑制。比 PPO 轻：不需要 value model、不需要 KL penalty 的完整实现。适合场景：reward 函数可计算（规则 reward），不需要复杂 critic 网络。

**项目结合：** train/train_grpo.py。


### 72. GRPO 为什么特别适合客服规则？

**面试回答**

客服规则可被写成可计算函数——must_include 覆盖加分、must_not_include 命中扣分、礼貌词加分、过度承诺扣分。GRPO 不需要 reward model，直接计算规则 reward。同一问题生成多条回答，好回答被强化——恰好匹配客服的多约束优化需求。

**项目结合：** train/rewards.py 的 客服规则奖励 函数。


### 73. 客服规则奖励（rewards.py）怎么设计的？

**面试回答**

多维奖励组合：base score=0.2（确保不完全为 0），must_include 覆盖率加 0-0.5，礼貌词加 0.15，最小长度（>=20 字）加 0.15，must_not_include 命中扣 0.5，风险表达扣 0.4。最终 clip 到 [0, 1]。设计原则：奖励是组合式的，防止单一维度被 hack。

**项目结合：** rewards.py 的 客服规则奖励 函数。参数：必须包含的内容（| 分隔的字符串）、禁止包含的内容。


### 74. reward hacking 是什么？怎么防止？

**面试回答**

模型学会最大化 reward 而非做好任务。如果 reward 只看关键词，模型堆关键词骗分。防止方法：多维 reward 防止单一维度被 hack；LLM Judge 和人工抽检作为 reward 外的质检。

**项目结合：** rewards.py 多维设计。evaluation.py + judge.py 做第二道质检。


### 75. 规则 reward 全 1.0 为什么不是好事？

**面试回答**

说明样例太简单——所有条款都被简单满足，模型不需要真正理解就能拿满 reward。应该设计有区分度的 eval 样本。


### 76. GRPO 会模板化吗？怎么解？

**面试回答**

会。如果 reward 单一，模型学固定套路。解决：多样化的训练数据 + 多维 reward + LLM Judge 质检。


### 77. 什么时候该训练 reward model 而不是规则 reward？

**面试回答**

当规则无法完全覆盖业务质量维度时——比如回答的"可信度"、"同理心"很难用关键词量化。有充足的高质量人工偏好数据后可以训练 reward model，再上 PPO。


## 八、PPO & RLHF

### 78. RLHF 标准三步骤？

**面试回答**

SFT → 用高质量数据微调出基础能力。Reward Model → 用偏好数据训练打分模型，让 chosen 得分高于 rejected。PPO → 用 reward model 优化 policy model，同时用 KL penalty 限制偏离参考模型过远。


### 79. 为什么项目不把 PPO 当主线？

**面试回答**

PPO 需要 reward model、value model、在线采样、KL 控制——工程复杂且不稳定。没有可靠 reward model 时 PPO 是空名词。第一阶段用 DPO/GRPO 更稳：DPO 离线简单，GRPO 可计算 reward。


### 80. PPO clipping 和 KL penalty 各有什么作用？

**面试回答**

Clipping 限制单步更新幅度——ratio 超出 [1-epsilon, 1+epsilon] 时梯度截断，防止单次更新崩。KL penalty 约束 policy 整体不偏离 reference model——reward 高不代表语言质量好，KL 保留基础能力。


### 81. 从 SFT 到 PPO 的合理顺序？

**面试回答**

SFT（学到任务格式）→ DPO（学到偏好）→ GRPO（学到规则约束）→ RM（学到精细打分）→ PPO（在线强化）。项目当前在 DPO+GRPO 阶段。


### 82. train_ppo.py 为什么只是 route checker？

**面试回答**

当前没有训练好的 reward model，跑 PPO 无意义。脚本只检查依赖和参数配置是否完整，确认"如果未来要跑 PPO，基础设施已就绪"。


## 九、评测体系

### 83. 项目的双层评测体系是哪两层？

**面试回答**

规则评测（关键词匹配，可复现）做快速回归，定位 must_include/must_not_include 问题。LLM Judge（语义理解）做多维质检，捕捉规则无法覆盖的问题。两层各输出独立评分，分歧挖掘层比较两者差异。


### 84. 规则评测的 7 个维度？

**面试回答**

answer_accuracy（综合准确）、policy_compliance（规则遵循）、completeness（完整性）、politeness（礼貌性）、safety（安全性）、forbidden_content（违禁内容率）、off_topic（答非所问率）。注意方向：forbidden_content 和 off_topic 在顶层是错误率（1=坏），在 dimensions 子 dict 中反转（1=好）。

**项目结合：** evaluation.py 的 score_case。AI_REVIEW_GUIDE 的指标方向一致性规则。


### 85. LLM Judge 的 rubric 怎么设计？

**面试回答**

固定 6 维度：accuracy（0-5，映射到 policy_compliance）、completeness（0-5）、politeness（0-5）、safety（0-5）、off_topic（true/false）、hallucination（true/false）。强制 JSON 输出，规避自由文本。温度设为 0.0 保证确定性。

**项目结合：** judge.py 的 build_judge_prompt。BL-001（score 归一化修复）。


### 86. Judge 的 accuracy 为什么映射到 policy_compliance 而不是 answer_accuracy？

**面试回答**

这是设计选择。LLM Judge 的"accuracy"实质是判断回答是否符合平台规则，和 rule 的 policy_compliance 概念上更接近。PGDM 的核心是让两个评测器测同一维度——所以 Judge 的 accuracy mapping 到 rule 的 policy_compliance。

**项目结合：** judge.py 的 normalize_judge_dimensions。


### 87. hallucination 维度为什么只在 Judge 里有？

**面试回答**

Hallucination 是语义理解任务——判断模型是否编造了规则或事实。规则评测基于关键词匹配无法判断"编造"。所以 hallucination 是 Judge 独有维度，不和规则评测对齐。

**项目结合：** AI_REVIEW_GUIDE 中记录了这是正确的设计。


### 88. LLM Judge 有什么已知偏差？

**面试回答**

Prompt 敏感——判词措辞变化影响打分。模型自我偏好——LLM 倾向于给自己的同类模型打高分。位置偏差——多回答对比时第一个或最后一个被偏好。缓解：固定 prompt、温度 0.0、随机顺序、人工抽检一致率。


### 89. DIMENSION_KEYS 怎么定义的？单方独有的维度怎么处理？

**面试回答**

DIMENSION_KEYS 在 disagreement.py 中定义，共 7 个共享维度。规则独有（如 forbidden_content）和 Judge 独有（如 hallucination）在分歧比较中静默跳过——只比较双方都有的维度。

**项目结合：** disagreement.py 的 DIMENSION_KEYS。AI_REVIEW_GUIDE 的"维度对齐"规则。


### 90. 客服场景的幻觉怎么定义？和通用 NLP 幻觉有何不同？

**面试回答**

客服幻觉：编造平台规则（"平台规定可以全额赔偿"而实际无此规则）、编造订单状态（"您的订单已在退款中"而实际未发起）、编造赔偿承诺（"今天一定到账"但平台无此流程）。通用 NLP 幻觉主要是事实错误，客服幻觉更危险——直接影响用户权益。


### 91. Preference Win Rate 怎么算？

**面试回答**

对同一组评测问题，让 LLM Judge 或人工盲评 Base/SFT/DPO/GRPO 各版本的回答。统计目标版本胜过 baseline 的比例。注意：随机顺序、匿名、重复测量消除位置偏差。比绝对分数更有说服力——说明模型确实在"更好"的方向上改进了。


### 92. 规则评测全 1.0 说明什么？

**面试回答**

说明样例太简单或 pipeline 刚跑通——但不代表模型强。规则评测是关键词匹配，全 1.0 可能只是评测集没有覆盖难度样本。正式实验需要 500-1000 条覆盖多场景、多难度的评测集。


### 93. 人工抽检怎么做？

**面试回答**

分层抽样——按场景（return_refund/logistics/safety_refusal 等）和 badcase 类型（over_promise/incomplete 等）各抽若干条。人审和 LLM Judge 结果对比算一致率。建议至少 100 条起步。


### 94. 指标提升但人工觉得差，听谁的？

**面试回答**

听人工，同时反查指标。说明指标没覆盖真正质量——可能 reward 被 hack，Judge 有偏好。把人工 badcase 回流到评测集固定问题，再更新指标定义。


### 95. 评测报告（report.py）包含哪些内容？

**面试回答**

整体指标（7 维度平均值）、分场景指标（每类 scenario 的 7 维度均值）、Badcase 统计（按类型分布）、Top-5 Badcase 示例。全部 Markdown 格式，可直接查看。

**项目结合：** report.py 的 build_markdown_report。

## 十、模型架构

### 96. Qwen2.5 是什么结构？

**面试回答**

Decoder-only causal LM，核心组件：RoPE 旋转位置编码、SwiGLU 门控 MLP、RMSNorm 归一化、GQA 分组查询注意力、Pre-norm 残差结构。Qwen2.5-7B-Instruct 有 28 层，Q heads=28，KV heads=4，使用 QKV bias。训练框架通过 LoRA 注入 q/k/v/o 和 gate/up/down 的线性层。


### 97. Qwen2.5 和 Qwen3 的核心架构差异？

**面试回答**

Qwen3 技术报告了两项主要变化：去掉 QKV bias（简化结构）、引入 QK-Norm（在 Q 和 K 上做 LayerNorm，稳定 attention logits）。主体结构（decoder-only、GQA、RoPE、SwiGLU、RMSNorm）不变。面试时一定要按版本回答，不混说。


### 98. Attention 公式？Q/K/V 分别做什么？

**面试回答**

公式 Attention = softmax(QK^T / sqrt(d_k)) * V。Q（Query）决定"当前 token 关注什么"；K（Key）决定"其他 token 提供什么信息"；V（Value）是被聚合的候选内容。sqrt(d_k) 是温度缩放因子，防止 softmax 进入极端梯度区。

**面试官视角**

**追问：手写？**
答：先写维度（b, h, seq, d_k），再写 mask（causal mask 防止看未来），最后说数值稳定性和计算优化。


### 99. MHA / MQA / GQA 的区别？

**面试回答**

MHA 每个 Q head 独享 K/V head，表达能力最强但 KV Cache 最大。MQA 所有 Q head 共享 1 个 K/V head，最省但可能损失表达。GQA（Qwen/Llama 2/3 使用）用 n 个 Q head 共享 m 个 K/V head（n>m>=1），在表达和效率之间折中。Qwen2.5-7B 是 28 Q heads 共享 4 KV heads，压缩比 7:1。

**项目结合：** 客服 prompt 可能很长，GQA 对长上下文推理更友好。


### 100. GQA 为什么省 KV Cache？

**面试回答**

KV Cache 缓存的是每层每个 KV head 的 K 和 V。KV heads 从 h（MHA）减少到 m（GQA），缓存量减少到 m/h。公式：cache_size ≈ layers * seq_len * 2 * n_kv_heads * head_dim * dtype_bytes。


### 101. KV Cache 显存怎么估算？

**面试回答**

公式：L * seq_len * 2（K+V）* n_kv_heads * head_dim * dtype_bytes。Qwen2.5-7B：28 层，KV heads=4，head_dim=128，fp16=2 字节。seq_len=2048 时：28*2048*2*4*128*2 ≈ 117MB。seq_len=8192 时约 470MB。多用户并发时乘以用户数。


### 102. RoPE 是什么？和绝对位置编码有什么区别？

**面试回答**

RoPE 不是在 embedding 上加位置向量，而是在 Q 和 K 上按位置施加旋转矩阵，使 attention 点积自然包含相对位置信息。绝对位置编码加固定位置向量，RoPE 旋转 Q/K。优点：RoPE 表达相对位置更自然——距离远的 token 间旋转角度大，QK 点积自然变小。长上下文时可通过 RoPE scaling（NTK-aware / YaRN）扩展。


### 103. SwiGLU 是什么？为什么比 ReLU 好？

**面试回答**

SwiGLU 是门控 MLP：output = Swish(xW_gate) * (xW_up) * W_down。Gate 分支控制信息通过量，表达力强于固定激活函数的 ReLU/GELU。计算量略大但效果更好，成为 Llama/Qwen 等现代 LLM 的标准选择。


### 104. RMSNorm vs LayerNorm？Pre-norm vs Post-norm？

**面试回答**

LayerNorm 减均值再除方差（需计算 mean 和 var）；RMSNorm 只按均方根缩放（不计算均值），计算更简单。Pre-norm 把 norm 放在子层之前——梯度更稳定，现代 LLM 都使用。Post-norm 在子层之后加 norm——训练更不稳定但表达能力稍强（原始 Transformer 方案）。


### 105. QK-Norm 是什么？

**面试回答**

在 Q 和 K 进入 attention 计算之前各做一次 LayerNorm，稳定 attention logits 的分布。Qwen3 引入，Qwen2.5 没有。


### 106. Tokenizer 对中文电商客服有什么影响？

**面试回答**

影响文本被模型看到的粒度和上下文长度。电商客服里有订单号（EC202606160001）、物流号、价格（¥159.00）、规格（A4/80g）、中英混合文本。如果 tokenizer 切碎了数字（如 "EC202606160001" 被切成 14 个 token），序列变长，学习和推理成本上升。清洗时不破坏关键符号很重要。


### 107. 为什么 LoRA 注入 Attention 和 MLP 的线性层，不注入 embedding 或 lm_head？

**面试回答**

embedding 和 lm_head 的参数量占比最大但和任务特定的"信息提取"关联较小。Attention 的 q/k/v/o 控制信息的关注和聚合方式，MLP 的 gate/up/down 控制表达能力——垂域适配主要靠这两个通路。


## 十一、推理与部署

### 108. vLLM 的 PagedAttention 解决什么？

**面试回答**

解决多请求推理时 KV Cache 显存碎片和低效管理问题。类似操作系统分页——把 KV Cache 切成固定大小的 block，通过 page table 管理，减少碎片、提高显存利用率和并发吞吐。长上下文和多并发场景收益明显。

**项目结合：** 项目部署时可用 vLLM 加载 LoRA adapter。


### 109. 推理显存大头在哪？多并发怎么管理 KV Cache？

**面试回答**

显存三大块：模型权重、KV Cache、中间激活。长文本多并发时 KV Cache 是最大头。管理策略：KV Cache 复用（prefix caching）、PagedAttention（分页管理）、量化（int8 KV Cache）。


### 110. 量化（int4/int8）会损失效果吗？

**面试回答**

可能——int4/int8 省显存但有精度损失，敏感任务（客服规则遵循）需测评确认。量化损失通常对生成质量影响有限，但会对需要精确记忆的 task 有影响。


### 111. adapter 推理怎么部署？分开 vs 合并？

**面试回答**

分开加载：基座模型不变，运行时指定加载哪个 adapter。便于多 adapter 切换、灰度、回滚。合并且权重后再部署简单（只需一个模型文件）但不灵活。客服场景推荐分开加载——规则经常变化，保留 adapter 版本管理。


### 112. 怎么减少生成延迟？四层优化？

**面试回答**

模型层：量化/小模型。输入层：规则摘要和截断。解码层：控制 max_new_tokens。系统层：batch、KV Cache、推理引擎（vLLM/TGI）。


### 113. 解码策略怎么选？为什么客服 temperature 要低？

**面试回答**

客服重准确合规——低 temperature（0.1-0.3）更合理。Greedy 最稳但死板，top-k/top-p 可以加一点多样性。温度高增加幻觉和过度承诺风险。评测时固定解码参数。


### 114. 长上下文对部署的挑战？

**面试回答**

显存增长（KV Cache 随序列长度线性增长）、延迟增加（计算量随序列长度二次增长）、位置外推（超过训练长度不稳定）、噪声增多（长上下文中无关信息干扰）。客服场景：做规则摘要和截断，prompt 不是越长越好。


### 115. 业务规则每天变，微调还合适吗？

**面试回答**

稳定话术和安全边界可以微调固化。每天变的具体规则（促销政策、运费）走 RAG 或规则库注入 prompt。EComCareLM 的 policy 字段已预留外部注入接口。


### 116. 项目如果要上线，整体架构怎么设计？

**面试回答**

规则/RAG 层提供上下文 → 微调模型生成回答 → 安全策略做后置校验 → 日志入 badcase 系统 → 定期回流训练。当前项目是训练对齐阶段，不是完整客服系统。


## 十二、Agent 面试题

### 117. 什么是 LLM Agent？核心组件？

**面试回答**

Agent = LLM（大脑）+ Planning（规划）+ Memory（记忆）+ Tools（工具）。LLM 做推理决策，Planning 做任务分解（ReAct/CoT），Memory 分短期（上下文窗口）和长期（向量数据库），Tools 通过 Function Calling/MCP 调用外部能力。

**项目结合：** bitalab 是 Agent 平台工程的实践——MCP 工具编排、Memory 管理、Multi-Agent 协作。


### 118. ReAct 框架的原理？ReAct vs CoT vs Act-Only？

**面试回答**

ReAct = Reasoning + Acting 交替。Thought 推理下一步要做什么 → Action 调用工具 → Observation 观察结果 → Thought 继续推理。CoT 只推理不行动，Act-Only 只行动不推理。ReAct 结合两者优势——推理指导工具使用，工具结果反馈改进推理。

**追问：和 Function Calling 是什么关系？**
答：ReAct 是框架层决定"要不要调用工具"，Function Calling 是实现层"怎么调用工具"。Agent 先用 ReAct 规划，再通过 Function Calling 执行。


### 119. Function Calling 的工作原理？

**面试回答**

工具定义预注册为 JSON schema（name、description、parameters）。LLM 判断需要调用工具时，输出结构化 JSON 包含函数名和参数。系统执行函数，结果返回给 LLM 整合为自然语言回复。


### 120. MCP 协议是什么？和 Function Calling 有什么区别？

**面试回答**

MCP（Model Context Protocol）由 Anthropic 提出，是连接 LLM 和外部工具/数据的标准化协议。Function Calling 是 API 级别每家厂商各自定义，MCP 是统一应用协议——一次实现、处处可用。MCP 支持动态工具发现（Tools）、资源读取（Resources）、提示模板（Prompts）。类比：Function Calling 是各品牌充电器，MCP 是 USB-C。

**项目结合：** bitalab 实现了 MCP Server/Client 架构。


### 121. Agent 记忆系统怎么设计？L1-L4 四层？

**面试回答**

L1 工作记忆（当前对话上下文窗口）。L2 短期记忆（滑动窗口+摘要）。L3 长期记忆（向量数据库存储用户偏好、关键信息）。L4 外部记忆（文档/知识库 RAG）。写入策略：判断重要性→选择存储层级。读取策略：相关性检索→时效性过滤→注入上下文。遗忘策略：基于时效性和重要性的衰减。

**项目结合：** bitalab 的 Memory 模块实现了长短记忆分层。


### 122. 多 Agent 协作有哪些模式？

**面试回答**

中心化（Orchestrator-Workers）：一个指挥者分配任务给多个 Worker，适合有明确流程的场景。去中心化（Peer-to-Peer）：各 Agent 自主协调，适合复杂动态场景。混合：中心化+去中心化结合。项目 bitalab 支持多种协作模式。


### 123. EComCareLM 和 bitalab 如果结合，会做成什么？

**面试回答**

bitalab 做 Agent 框架和工具编排，EComCareLM 提供垂域微调模型。结合就是电商客服专有 Agent——用户在 bitalab 创建对话，背后模型是 EComCareLM 微调版本，遇到规则查询通过 MCP tool 检索。这是两个项目的自然延伸。


### 124. Tool Calling 的幻觉怎么处理？

**面试回答**

参数幻觉（LLM 填错工具参数）和工具选择幻觉（LLM 选错工具）。缓解：约束解码（强制匹配合法参数）、验证层（执行前参数校验）、Retry 机制（失败后重选工具）。项目 bitalab 在 Runner 层有参数校验。


### 125. Agent 评测怎么做？

**面试回答**

Tool Call 准确率（工具选对且参数正确）、任务完成率（端到端目标达成）、鲁棒性（中断恢复、异常输入）。Agent 评测比纯文本生成更复杂——不仅要看最终结果，还要看中间规划路径。

**项目结合：** EComCareLM 的评测方法（规则+Judge）可以扩展到 Agent 场景。


### 126. 2025-2026 年 Agent 技术趋势？

**面试回答**

MCP 协议标准化（工具集成统一）、A2A（Agent-to-Agent 通信标准）、Computer Use（Agent 直接操控 GUI）、Long-running Agent（持续运行数小时甚至数天的自治 Agent）、多模态 Agent（视觉+语音+文本）。
